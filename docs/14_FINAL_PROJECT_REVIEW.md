# WOONG AI 프로젝트 최종 결과 및 회고

## 1. 최종 아키텍처 요약

### 1.1 전체 구조

WOONG AI는 창고 운영 데이터를 관리하는 SQLite 기반 WMS DB(`wms.db`)를 운영 상태의 단일 원천으로 사용하고, 다음 세 계층을 하나의 웹 애플리케이션으로 통합한 Agentic 서비스이다.

1. **대화형 운영 Copilot**
   - 자연어 질의를 업무 Intent로 분류한다.
   - 운영 데이터를 조회하거나 결정론적 업무 Tool을 실행한다.
   - 정책·SOP 근거가 필요한 경우 RAG를 수행한다.
   - 고위험 상태 변경은 Draft를 생성하고 사용자 승인을 요구한다.

2. **Blackboard 멀티에이전트 자동운영**
   - 주기적으로 운영 이벤트를 수집한다.
   - 도메인 에이전트가 실행 가능한 Action을 제안한다.
   - 우선순위와 자원·정책·공간 제약을 검증한 후 Action을 실행한다.
   - 작업팀 배정과 Zone 작업 진행을 자동으로 조정한다.

3. **분석 및 시각화**
   - DES 기반 창고 운영 시뮬레이션과 What-if 분석을 제공한다.
   - 2D 디지털 트윈으로 작업 흐름을 시각화한다.
   - KPI 대시보드에서 현재 운영 상태와 목표 대비 수준을 조회한다.

운영 데이터와 상태는 SQLite에서 관리하고, 정책·SOP 지식은 별도 문서와 FAISS 인덱스로 관리한다. 따라서 운영 사실은 DB에서, 정책적 설명은 검색 근거에서 가져오도록 역할을 분리했다.

### 1.2 최종 산출물

최종 산출물은 다음 기능을 하나의 화면에서 제공하는 웹 기반 창고 운영 Copilot이다.

- Agent Chat
- KPI Dashboard
- 운영 데이터 조회
- Warehouse Simulation 및 2D 디지털 트윈
- Approval
- AI 동작 추적
- 창고 자동운영

백엔드는 FastAPI로 구성했으며, SSE를 통해 LangGraph 노드와 세부 실행 단계의 진행 상태를 실시간으로 전달한다. 최종 답변은 그래프 실행이 끝난 후 전달된다.

초기 운영 데이터는 시드 생성기로 재현 가능하게 구성하며, 평가 실행 시에는 라이브 DB와 분리된 fresh-seed 임시 DB를 사용한다.

### 1.3 대화형 Agent 구조

```text
사용자 질의
  → FastAPI / SSE
  → Router
  → Parameter Extractor·Validator
  → Rule-based Planner
  → Deterministic Tool Executor
  → Verifier
  → RAG Decision
  → 필요 시 Retriever
  → Response Generator
  → Approval Gate
```

각 단계의 역할은 다음과 같다.

- **Router**
  - 경량 LLM을 사용해 26종 Intent와 초기 파라미터를 분류한다.

- **Parameter Extractor·Validator**
  - Intent별 필수 파라미터가 존재하는지 검증한다.
  - 필수값이 부족하면 실행하지 않고 사용자에게 추가 정보를 요청한다.

- **Planner**
  - 분류된 Intent를 기준으로 실행할 업무 경로를 규칙 기반으로 결정한다.

- **Tool Executor**
  - `_HANDLERS[intent]` 매핑을 사용해 실행할 핸들러를 결정한다.
  - LLM은 의도와 파라미터를 해석하고, DB 조회·계산·Draft 생성은 코드가 담당한다.
  - 이를 통해 LLM이 임의의 업무 함수나 SQL을 직접 실행하지 못하도록 제한한다.

- **Verifier**
  - 현재 구현에서는 Tool 결과에 대한 규칙 검증을 수행한다.
  - 특히 적치 추천 점수의 정규화 범위 등을 확인한다.
  - 다단계 재계획 루프는 향후 확장 대상으로 남아 있다.

- **RAG Decision·Retriever**
  - 정책, SOP, 추천 근거가 필요한 Intent에만 RAG를 수행한다.
  - 검색 후보를 PRISM 방식으로 재평가한다.
  - Sufficient Context 게이트로 근거의 충분성을 판단한다.
  - 근거가 부족하면 query rewrite 후 최대 2회 재검색한다.
  - 최종적으로 근거가 충분하지 않으면 답변을 생성하지 않고 abstain한다.

- **Response Generator**
  - Tool 결과와 RAG 근거만 사용해 최종 답변을 생성한다.
  - 근거 없는 수치나 정책을 임의로 생성하지 않도록 프롬프트와 평가 하네스로 통제한다.

- **Approval Gate**
  - 적치지시, 피킹지시, 출고확정, 발주와 같은 고위험 변경 Intent는 실제 업무 상태를 즉시 변경하지 않는다.
  - Tool Executor에서 Draft만 생성하고, Approval Gate가 승인 필요 상태와 Draft 정보를 반환한다.
  - 실제 변경은 사용자가 승인 API를 호출한 이후에만 수행된다.

### 1.4 Blackboard 자동운영 구조

```text
Control Loop
  → 신규 이벤트 수집
  → 도메인 에이전트 Action 제안
  → 우선순위 정렬
  → Simulation / Policy / Lock 게이트
  → Executor
  → ZoneScheduler
  → 실행 결과 및 계산 근거 기록
```

자동운영의 주요 특징은 다음과 같다.

- `action_type`별 기본 우선순위와 개별 `priority_score`를 결합해 실행순서를 결정한다.
- ZoneScheduler는 자원 해제를 우선하기 위해 다음 순서로 동작한다.

```text
FINISH_ZONE_LEG
  → START_ZONE_WORK
  → ALLOCATE_TEAM
```

- 작업팀 배정 후보는 `dispatch_score`를 기준으로 정렬한다.
- 피킹 작업은 입구에서 출발해 필요한 Zone을 방문하고 다시 입구로 돌아오는 closed TSP 경로를 계산한다.
- 자동 할당과 보충처럼 승인 없이 실행되는 저위험 Action은 Policy·Lock·자원 제약·멱등성 검사를 적용한다.
- 배정 점수, 방문 경로, 실행순서를 다음 테이블에 저장한다.

| 기록 대상 | 저장 테이블 |
| --- | --- |
| 작업 배정 점수와 인수 | `dispatch_scores` |
| Zone 방문순서와 경로 | `zone_routes` |
| Action 실행순서와 결과 | `action_exec_log` |

### 1.5 관측 가능성

각 대화 요청의 실행 정보는 `agent_traces`에 저장한다.

- 분류된 Intent와 신뢰도
- 노드별 실행 결과
- RAG 수행 및 충분성 판단 결과
- 재검색 횟수와 abstain 여부
- 사용 근거
- 승인 필요 여부와 Draft
- LLM 호출 수
- 입력·출력·전체 토큰 수
- 최종 응답

이를 통해 기능 오류뿐 아니라 비용 증가, 호출 수 증가, 근거 품질 저하와 같은 품질 회귀도 추적할 수 있다.

### 2.1 성능 측정 결과

측정 조건은 warm cache, 동일 입력, 유형별 3회 평균, 단일 로컬 서버, 사내 LLM 게이트웨이이다.

| 질의 유형 | 총지연 | LLM 호출 | 토큰 | 추정 게이트웨이 구간 | 차감 후 잔여시간 |
| --- | ---: | ---: | ---: | ---: | ---: |
| KPI 조회 | 12.82초 | 8회 | 8,879 | 7.07초 | 5.75초 |
| 정책 질의 | 7.94초 | 5회 | 5,628 | 4.42초 | 3.52초 |
| 재고위험 예측 | 15.96초 | 8회 | 8,490 | 7.07초 | 8.88초 |

전체 측정 지연은 약 7.9~16.0초이며, 조회·예측형은 12.8~16.0초이다. 호출당 최소 왕복시간 약 0.88초를 기준으로 추정하면 다수의 순차 LLM 호출이 지연의 큰 부분을 차지한다.

다만 ‘차감 후 잔여시간’은 총지연에서 최소 왕복시간과 호출 수의 곱을 뺀 추정치이다. 따라서 이를 정확한 CPU 처리시간이나 완전한 순수 애플리케이션 처리시간으로 해석해서는 안 된다.

### 2.2 평가 결과 해석 시 주의사항

- 60/60은 현재 정의된 자동 검증 항목에 대한 통과율이다.
- Intent 15/15는 26개 Intent 전체의 완전한 커버리지를 의미하지 않는다.
- RAG 4/4 중 실제 근거 부족 사례는 1건이다.
- Faithfulness와 Relevance는 각각 5개 질의를 대상으로 측정했다.
- Judge 판별력은 4개 negative control로 확인했다.
- 따라서 100%라는 값은 모집단 성능이 아니라 현재 평가 표본에서 실패가 없었다는 의미로 해석해야 한다.

## 3. 제공 가치 및 기대 효과

### 3.1 비즈니스 가치

- 입고·적치·피킹·출고 작업의 우선순위를 정형화해 운영자의 반복적인 판단 부담을 줄일 수 있다.
- KPI 이상이나 정책 문의에 운영 데이터와 SOP 근거를 함께 제공해 원인 확인과 자료 탐색 시간을 단축할 수 있다.
- 작업팀·지게차·입고 지연·수요 변화 등에 대한 What-if 분석으로 자원 투자 전 예상 효과를 비교할 수 있다.
- 배정 점수, 경로, Action 실행순서를 기록하므로 “왜 이 작업이 먼저 처리됐는가”를 사후 검증할 수 있다.
- 운영자 승인과 자동화 영역을 구분해 단순 조회형 Copilot을 실제 업무 Action으로 연결할 수 있다.

다만 시간 절감률, 처리량 증가율, 비용 절감액과 같은 사업 성과는 아직 실운영 전후 비교로 측정하지 않았다. 현재 결과는 해당 효과를 측정할 수 있는 기능과 데이터 기반을 확보했다는 의미이다.

### 3.2 기술적 가치

- 자연어 해석과 업무 실행을 분리해 LLM의 역할을 Intent·파라미터 해석과 설명 생성으로 제한했다.
- 실행 가능한 Tool을 코드 레지스트리에 명시해 임의 함수 호출과 임의 SQL 생성을 방지했다.
- 고위험 변경은 Draft와 HITL 승인 게이트를 통과하도록 구성했다.
- RAG에 검색 결과 재평가, 충분성 판단, 재검색, abstain을 결합해 근거 없는 답변을 억제했다.
- 이벤트 기반 Blackboard 구조에 우선순위, 정책, 락, 자원 제약과 멱등성 통제를 결합했다.
- `dispatch_score`와 closed TSP를 외부 최적화 라이브러리 없이 구현해 배정 및 이동순서의 계산 근거를 남겼다.
- fresh-seed 격리 DB를 사용하는 평가 하네스로 운영 데이터와 테스트 데이터를 분리했다.
- LLM-as-a-Judge에 negative control을 포함해 평가 모델 자체의 판별력을 함께 확인했다.
- 요청별 호출 수, 토큰, 근거와 승인 여부를 기록해 비용·품질 회귀를 분석할 수 있게 했다.

## 4. 운영 및 보안 고려사항

### 4.1 인증과 사용자 권한

현재 POC는 세션 기반 단일 사용자 환경을 전제로 한다. 멀티유저 인증, 역할 기반 권한, 사용자별 승인 귀속과 책임 분리는 구현 범위에 포함하지 않았다.

운영 환경으로 확장할 때는 다음 기능이 필요하다.

- 사용자 인증
- 역할 기반 또는 속성 기반 권한 통제
- Draft 생성자와 승인자 귀속
- 중요 Action의 생성자·승인자 분리
- 세션 만료 및 재인증
- 승인 이력의 장기 보존

### 4.2 상태 변경 통제

고위험 상태 변경 4종은 즉시 실행하지 않는다.

- 적치지시 생성
- 피킹지시 생성
- 출고확정
- 발주 생성

이들 Action은 Draft를 생성한 뒤 사용자 승인을 받아야 실제 업무 상태가 변경된다.

자동 할당과 자동 보충 등 일부 저위험 변경은 Approval Gate 대상에서 제외되어 있다. 대신 다음 통제를 적용한다.

- 허용된 Action 유형만 실행하는 코드 기반 allowlist
- Policy Gate
- Resource 및 Zone Lock
- 용량·자원 제약 검증
- `idempotency_key`를 통한 중복 실행 방지
- Action 실행 로그 및 감사 기록

따라서 “모든 상태 변경이 HITL을 거친다”기보다는, Action 위험도에 따라 승인형과 정책 기반 자동실행형으로 구분한 구조이다.

### 4.3 프롬프트 인젝션 대응

프롬프트 인젝션 대응은 Router 하나에만 의존하지 않는다.

```text
입력 범위 분류
  → 고정 Intent 집합
  → 파라미터 검증
  → 결정론적 Handler allowlist
  → 상태 변경 Draft
  → Approval Gate
```

공격성 입력이나 시스템 프롬프트 노출 요청은 `out_of_scope` 또는 비실행 대화 경로로 분리된다. 설령 Intent 분류가 잘못되더라도 Tool Executor는 등록된 Handler만 실행할 수 있으며, 고위험 변경은 승인 없이 완료되지 않는다.

평가 하네스에서는 인젝션 4건과 승인 우회 4건에 대해 업무 상태가 무승인으로 변경되지 않음을 확인했다. 이는 현재 테스트 사례에 대한 검증 결과이며, 모든 형태의 공격을 방어한다는 의미는 아니다.

### 4.4 장애 대응

- 자동운영 시작 후 DES 워밍업이 완료되지 않은 경우 이벤트 상태를 보존하고 해당 사이클의 처리를 대기한다.
- DES가 오류 결과를 반환한 경우에는 영구적인 자동운영 중단을 방지하기 위해 fail-open으로 진행한다.
- RAG 근거가 부족하면 답변을 생성하지 않고 abstain한다.
- Zone 단일 점유, 자원 가용성, 정책 및 락 게이트로 동시 실행 충돌을 방지한다.
- 멱등성 키를 이용해 동일 Action의 중복 실행을 억제한다.

DES fail-open은 가용성을 우선한 POC 정책이다. 안전이 중요한 도메인에서는 Action 위험도에 따라 fail-closed 또는 수동 전환 정책을 적용해야 한다.

### 4.5 운영 전 추가 보안 과제

- LLM 게이트웨이 키와 DB 접근정보의 비밀 관리
- 로그 내 개인정보와 민감 업무정보 마스킹
- API별 인증·인가
- 승인 요청 위·변조 방지
- 감사 로그의 변경 불가능성 보장
- 입력 크기, 호출 빈도, 세션별 사용량 제한
- 모델 및 정책 문서 버전별 실행 이력 관리

## 5. 회고 및 범용 도메인 확장 방향

### 5.1 일반화 가능한 핵심 설계 원칙

이 프로젝트에서 가장 일반화하기 쉬운 부분은 창고 도메인의 개별 규칙이 아니라, 자연어 요청을 안전한 업무 실행으로 변환하는 구조이다.

#### 원칙 1. 언어 해석과 업무 실행을 분리한다

LLM은 다음 역할에 집중한다.

- 사용자 의도 분류
- 파라미터 추출
- 검색 문서 재평가
- 근거 기반 설명 생성

실제 업무 실행은 다음 요소가 담당한다.

- 명시적으로 등록된 Tool Handler
- 스키마 및 파라미터 검증
- 업무 규칙
- 트랜잭션
- 멱등성 제어

이 구조는 제조, 물류, 고객지원, 구매, 금융 백오피스 등 상태 변경이 수반되는 대부분의 업무 Agent에 적용할 수 있다.

#### 원칙 2. Action을 위험도에 따라 분류한다

모든 Action에 동일한 승인 정책을 적용할 필요는 없다.

| 위험 수준 | 처리 방식 | 예시 |
| --- | --- | --- |
| 조회 | 즉시 실행 | 현황·KPI·이력 조회 |
| 추천 | 즉시 계산, 근거 제공 | 우선순위·대응방안 추천 |
| 저위험 변경 | 정책 기반 자동실행 | 자동 할당·내부 보충 |
| 고위험 변경 | Draft 및 HITL 승인 | 확정·발주·대외 반영 |
| 금지 Action | 실행 경로 미제공 | 권한 없는 삭제·정책 우회 |

다른 도메인으로 이전할 때는 개별 Action을 이 분류에 매핑하고 승인, 자동실행, 금지 정책을 결정하면 된다.

#### 원칙 3. 운영 상태와 정책 지식을 분리한다

- 현재 상태와 거래 사실은 정형 DB에서 조회한다.
- 규정, SOP, 산식과 설명 근거는 문서 검색으로 조회한다.
- 답변 생성 시 두 근거를 함께 사용한다.

이 구조는 운영 데이터의 최신성과 정책 설명의 추적 가능성을 동시에 확보한다. 다만 정책 문서와 DB가 서로 다른 버전이나 기준시점을 사용하지 않도록 버전 관리가 필요하다.

#### 원칙 4. 근거가 부족하면 답하지 않는다

RAG는 단순 유사도 검색에서 끝나지 않고 다음 단계를 포함해야 한다.

```text
후보 검색
  → 재평가
  → 근거 충분성 판단
  → 질의 재작성 및 재검색
  → 근거 반환 또는 abstain
```

의료, 법무, 금융, 품질관리처럼 잘못된 답변의 비용이 큰 도메인일수록 이 원칙이 중요하다.

#### 원칙 5. 자동운영의 모든 판단 근거를 기록한다

자동화 결과만 저장하면 사후 설명이 어렵다. 다음 정보를 함께 남겨야 한다.

- 후보 목록
- 점수 구성요소
- 적용 정책
- 우선순위
- 제외 사유
- 사용한 자원
- 실행순서
- 최종 결과

이 원칙을 적용하면 설명 가능성, 감사, 장애 분석과 가중치 튜닝에 동일한 로그를 사용할 수 있다.

#### 원칙 6. 평가 하네스를 기능의 일부로 취급한다

Agent 평가는 단순 답변 정확도만으로 충분하지 않다.

- Intent 및 파라미터 정확도
- Tool 선택과 결정성
- 업무 규칙 준수
- 근거 충분성 및 abstain
- 무승인 상태 변경
- 인젝션과 우회 시도
- LLM Judge의 판별력
- 지연시간, 호출 수와 토큰
- 시뮬레이션 재현성

이 항목을 고정 데이터와 기대값으로 자동화해야 도메인이 변경되거나 기능이 확장되어도 회귀를 확인할 수 있다.

### 5.2 다른 도메인에서도 재사용할 수 있는 영역

| 재사용 가능한 코어 | 도메인별 교체·확장 영역 |
| --- | --- |
| Router–Tool–Verifier 실행 구조 | Intent 목록과 업무 용어 |
| Tool Handler 레지스트리 | 업무별 Handler 구현 |
| Draft 및 Approval Gate | Action별 위험등급과 승인 정책 |
| Blackboard 이벤트·Action 구조 | 이벤트 유형과 도메인 에이전트 |
| Policy·Lock·멱등성 게이트 | 업무 제약과 동시성 규칙 |
| 우선순위 실행 프레임워크 | 점수 항목과 가중치 |
| RAG 충분성 및 abstain | 정책·SOP 문서와 메타데이터 |
| Trace와 감사 로그 | 도메인별 감사 필드 |
| 평가 하네스 구조 | 평가 질의, 기대값과 위험 시나리오 |
| 시뮬레이션 실행·비교 구조 | 도메인별 공정·자원·확률 모델 |

예를 들어 제조 도메인에서는 Zone을 설비나 공정으로, 작업팀을 작업자·장비로, 출고 마감을 생산 납기로 바꿀 수 있다. 고객지원에서는 이벤트를 고객 티켓으로, Action을 담당자 배정·에스컬레이션·환불 승인으로 치환할 수 있다.

### 5.3 일반화 관점에서 확인된 기술적 한계

#### 다단계 LLM 호출로 인한 지연

현재 지연은 개별 모델의 생성시간뿐 아니라 순차적인 LLM 호출 수에 크게 영향을 받는다. 이 문제는 LLM 기반 업무 Agent 전반에서 공통으로 발생한다.

개선 방향은 다음과 같다.

- Router와 파라미터 추출 통합
- RAG가 불필요한 질의의 조기 종료
- 독립적인 판정 단계의 병렬 실행
- 동일 정책 질의에 대한 검색·리랭크 캐싱
- 프롬프트 및 대화 컨텍스트 축소
- 단계별 모델 크기 차등 적용
- 충분성 점수가 높은 경우 재검색 생략
- 구조화 출력 실패 시에만 LLM 재호출

#### 단일 파일 DB의 확장성

SQLite는 POC와 재현 가능한 평가에는 적합하지만 다음 조건에서는 한계가 있다.

- 다수 사용자의 동시 상태 변경
- 여러 자동운영 Worker의 병렬 실행
- 장기 실행 트랜잭션
- 고가용성과 장애 복구
- 대규모 운영 이력 조회

확장 시에는 PostgreSQL 등 트랜잭션형 RDBMS로 이전하고 다음 패턴을 함께 검토해야 한다.

- 낙관적 또는 비관적 잠금
- Outbox/Event Queue
- 트랜잭션별 멱등성
- Action 상태 머신
- 읽기·쓰기 부하 분리
- 테스트 DB 자동 격리

#### 인증과 승인 책임의 귀속

세션만으로는 누가 Action을 요청하고 승인했는지 책임을 명확히 할 수 없다. 범용 업무 서비스로 확장하려면 인증 주체, 조직, 역할과 승인 권한을 Draft 및 감사 로그에 연결해야 한다.

#### 도메인 용어와 멀티턴 라우팅 편차

“이 용어가 무엇인가”와 같은 질의는 대화 맥락 설명과 정책 문서 검색 사이의 경계가 모호하다. 이는 창고뿐 아니라 전문 용어가 많은 모든 도메인에서 발생한다.

개선 방향은 다음과 같다.

- 도메인 용어 사전과 동의어 사전 구축
- 정책 용어 Intent 또는 glossary Intent 분리
- 질문 유형뿐 아니라 답변에 필요한 근거 유형을 함께 분류
- 낮은 분류 신뢰도에서 확인 질문 수행
- 멀티턴 문맥이 있더라도 정책 용어는 RAG로 교차 검증
- 실제 실패 로그를 Intent 평가 세트에 지속적으로 추가

#### 평가 표본과 외적 타당성

현재 품질 평가 표본은 기능 성립을 확인하기에는 유효하지만, 운영 환경의 전체 분포를 대표하기에는 작다.

범용화를 위해서는 다음과 같은 평가 확대가 필요하다.

- Intent별 최소 표본 수 설정
- 전체 Intent 라벨 커버리지 측정
- 정상·경계·공격·오타·축약 질의 분리
- 단일 턴과 멀티턴 평가 분리
- 여러 seed와 데이터 규모에서 재현성 검증
- Judge 모델·프롬프트 변경에 대한 민감도 분석
- 실제 사용자 로그 기반 익명화 평가 세트
- 운영 적용 전 shadow mode 평가
- 실운영 결과와 추천 결과의 사후 비교

#### 최적화 알고리즘의 규모 한계

현재 closed TSP는 Zone 수가 작은 환경에서 정확한 경로를 계산하기에 적합하다. 그러나 완전탐색 방식은 방문 대상 수가 증가하면 계산량이 급격히 증가한다.

규모가 커지면 다음 방식으로 교체할 수 있다.

- 최근접 이웃 및 2-opt 휴리스틱
- 제한시간 기반 탐색
- 그래프 최단경로
- 제약계획 또는 정수계획 Solver
- 과거 실행 결과를 이용한 근사 모델

중요한 점은 알고리즘을 교체하더라도 입력, 출력, 점수 근거와 실행 로그의 계약을 유지하는 것이다.

#### 시뮬레이션과 현실 간 오차

시뮬레이션은 입력 파라미터와 공정 가정의 품질에 영향을 받는다. 다른 도메인으로 이전할 때는 모델을 그대로 재사용하기보다 시뮬레이션 실행·버전·비교 구조를 재사용하고, 공정 모델은 새 도메인 데이터로 다시 정의해야 한다.

- 실제 처리시간 분포 추정
- 예측값과 실제값의 오차 기록
- 파라미터 버전 관리
- 시나리오별 신뢰구간 제공
- 모델 변경 전후 결과 비교
- 데이터 부족 시 예측 사용 제한

### 5.4 단계별 확장 로드맵

#### 1단계: 안전성과 운영 기반 강화

- 멀티유저 인증 및 역할 기반 권한 도입
- Action별 위험등급과 승인 정책 명문화
- 생성자·승인자 분리
- 승인 및 감사 로그의 무결성 강화
- fail-open·fail-closed 정책을 Action 위험도별로 설정
- PostgreSQL 기반 동시성 구조 설계

#### 2단계: 검색과 대화 품질 개선

- BM25와 임베딩을 결합한 Hybrid Search 도입
- 정책 문서 메타데이터 및 버전 관리
- 도메인 용어·동의어 사전 구축
- 용어 정의 질의를 정책 검색으로 연결
- 낮은 신뢰도에서 확인 질문 수행
- RAG 검색·리랭크·충분성 평가 표본 확대

#### 3단계: 성능 최적화

- Router와 파라미터 추출 단계 통합
- 불필요한 RAG 및 LLM 호출 제거
- 프롬프트 캐싱과 검색 결과 캐싱
- 경량 모델과 생성 모델의 역할 분리
- 독립 단계 병렬화
- 노드별 지연시간과 호출 비용 예산 설정

#### 4단계: 데이터 기반 자동운영 고도화

- 운영 로그로 우선순위 가중치 보정
- 처리시간 파라미터를 실측 분포로 교체
- 추천 결과와 실제 처리 결과 비교
- 공정성, 마감 준수, 이동거리와 가동률을 함께 고려하는 다목적 최적화
- 현재의 완전탐색 TSP를 확장 가능한 경로 최적화 방식으로 교체
- 자동화 Action별 성과와 오류율 모니터링

#### 5단계: 평가와 배포 체계 확장

- 전체 Intent를 포함하는 회귀 세트 구축
- 데이터 규모와 동시 사용자 수별 부하 테스트
- 모델·프롬프트·정책 버전별 비교 평가
- 배포 전 평가 하네스 자동 실행
- shadow mode와 단계적 자동화 확대
- 품질 기준 미달 시 이전 버전으로 전환할 수 있는 운영 절차 구축

### 5.5 최종 회고

이 프로젝트에서 확인한 핵심은 Agentic 서비스의 품질이 LLM 성능만으로 결정되지 않는다는 점이다. 실제 업무에 적용하려면 다음 요소가 함께 설계되어야 한다.

```text
자연어 이해
  + 결정론적 업무 실행
  + 근거 충분성 검증
  + 위험도 기반 승인
  + 정책·락·멱등성 통제
  + 실행 근거 기록
  + 재현 가능한 평가
```

WOONG AI의 창고 데이터 모델, 배정 산식과 시뮬레이션 모델은 도메인 전용 요소다. 반면 Router–Tool–Verifier 구조, RAG 충분성 게이트, Draft 승인, Blackboard Action 처리, Trace 및 평가 하네스는 다른 운영 도메인에도 재사용할 수 있는 공통 기반이다.

따라서 다른 도메인으로 확장할 때는 전체 시스템을 새로 구축하기보다 공통 실행·통제 계층을 유지하고, Intent, 데이터 모델, Tool, 정책 문서, 우선순위 산식과 시뮬레이션 모델을 해당 도메인에 맞게 교체하는 방식이 적절하다.
