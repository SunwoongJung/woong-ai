# WOONG AI 5분 시연 영상 스크립트

## 1. 시연 콘셉트

이 영상의 핵심 메시지는 다음 한 문장이다.

> WOONG AI는 창고 현황을 설명하는 챗봇이 아니라, 운영 데이터와 정책 근거를 바탕으로 판단하고, 승인과 통제 아래 실제 업무를 실행하며, 그 전 과정을 추적할 수 있는 Agentic 운영 서비스입니다.

기능을 탭별로 단순 나열하지 않고 다음 업무 흐름을 하나의 이야기로 연결한다.

```text
운영 현황 확인
  → AI 적치 추천
  → 상태 변경 Draft 생성
  → Agent 추론·실행 Trace 검증
  → 사용자 승인 및 실제 반영
  → Blackboard 자동운영 확인
  → What-if 시뮬레이션
```

## 3. 상세 시연 스크립트

### 00:00~00:20 — 오프닝 및 서비스 개요

**화면 조작**

1. KPI Dashboard를 전체 화면으로 보여준다.
2. 상단 탭과 KPI 카드, Zone 점유율, 작업팀 가동률 영역을 천천히 가리킨다.
3. 마우스를 빠르게 흔들지 않고 핵심 카드 위에서 잠시 멈춘다.

**내레이션**

> 웨어하우스 옵스 오케스트레이션 앤 네비게이션 거버넌스 AI, 일명 WOONG AI는 대화형 Copilot, 멀티에이전트 자동운영, 디지털 트윈과 시뮬레이션을 하나로 통합한 창고 운영 서비스입니다. 
모든 화면은 동일한 WMS 데이터에서 현재 상태를 조회하므로, 질문과 실행, 예측 결과가 하나의 운영 흐름으로 연결됩니다.

**강조 포인트**

- 단순 챗봇이 아닌 단일 웹 기반 운영 서비스
- 대화, 실행, 자동운영, 분석이 동일한 운영 상태를 공유

### 00:20~01:00 — Agent Chat에서 근거 기반 적치 추천

**화면 조작**

1. `Agent Chat` 탭으로 이동한다.
2. 새 대화에서 다음 질문을 입력한다.

```text
INB003 적치 위치 추천해줘
```

3. 응답을 기다리는 동안 화면에 순차적으로 표시되는 Router, Tool, RAG 등의 실행 단계를 가리킨다.
4. 응답이 완료되면 추천 Location, 점수, 동일 SKU·잔여용량·거리 등의 추천 근거를 보여준다.
5. RAG 출처가 표시되면 정책 문서 출처도 가리킨다.

**내레이션**

> 먼저 입고건 INB003의 적치 위치를 자연어로 요청하겠습니다. 
요청은 LangGraph를 따라 Intent 분류, 파라미터 검증, Tool 실행과 정책 검색을 거칩니다. 
여기서 LLM이 위치를 임의로 결정하는 것이 아니라, 실제 재고와 Location 용량을 코드로 계산하고 정책 문서를 근거로 설명합니다.

응답이 표시된 뒤:

> 추천 결과에는 위치만 있는 것이 아니라 점수와 계산 근거가 함께 제공됩니다. 즉 운영자는 무엇을 추천했는지뿐 아니라 왜 이 위치인지까지 확인할 수 있습니다.

**AI 구축 관점 강조**

- Router는 자연어를 Intent와 파라미터로 구조화
- 실제 계산은 결정론적 Tool Handler가 수행
- RAG는 정책·SOP 설명에만 사용
- 생성 모델과 업무 계산의 역할 분리

**서비스 관점 강조**

- 운영자가 별도 화면에서 재고와 용량을 일일이 대조할 필요가 없음
- 추천과 판단 근거를 한 화면에서 확인

### 01:00~01:30 — 상태 변경 요청과 Draft 생성

**화면 조작**

1. 같은 채팅에서 다음 요청을 입력한다.

```text
INB003를 L-A-001에 적치지시 생성해줘
```

2. 답변 아래에 나타나는 승인 필요 카드와 Draft ID를 보여준다.
3. 아직 실제 상태가 변경되지 않았다는 안내를 가리킨다.
4. 이 화면에서는 승인하지 않고 `Approval` 탭에서 처리하기 위해 보류하거나 그대로 둔다.

**내레이션**

> 이제 추천을 실제 적치지시로 연결하겠습니다. 
상태를 바꾸는 요청이지만 즉시 실행되지는 않습니다. 
시스템은 실행 전 검증 결과를 Draft로 만들고 Approval Gate를 통해 사용자 승인 대기 상태로 전환합니다.

> 자연어로 업무를 실행할 수 있으면서도, 고위험 변경은 반드시 사람이 최종 통제하도록 설계했습니다.

**강조 포인트**

- Actionable Agent
- 조회와 상태 변경 경로 분리
- Draft 기반 Dry-run
- HITL 승인으로 오작동과 승인 우회 방지

---

### 01:30~02:35 — AI 관측에서 Agent 추론·실행 로그 확인

이 장면은 전체 영상에서 가장 중요한 구간이다. 최소 60초를 확보한다.

**화면 조작**

1. `AI 관측` 탭으로 이동한다.
2. 방금 생성된 최신 Trace를 선택한다.
3. 상태 변경 Trace에서 다음 항목을 순서대로 펼친다.
   - Router
   - Parameter Extractor
   - Planner
   - Tool Executor
   - Verifier
   - Response Generator
   - Approval Gate
4. Router에서 `stocking_task_create`, `inbound_no=INB003`, `location_id=L-A-001`을 가리킨다.
5. Tool Executor에서 생성된 Draft 결과를 보여준다.
6. Approval Gate에서 `승인 필요`와 Draft ID를 가리킨다.
7. 상단 또는 Trace 요약에 표시되는 전체 토큰 수와 LLM 호출 수를 가리킨다.
8. 직전 적치 추천 Trace도 선택한다.
9. RAG Retriever의 다음 세부 로그를 펼쳐 보여준다.
   - 문서 검색
   - PRISM 리랭크
   - Sufficient Context 판정
   - 사용된 근거 문서

**내레이션**

> AI 관측 화면에서는 최종 답변만 보는 것이 아니라, Agent가 요청을 어떻게 해석하고 실행했는지 전체 Trace를 확인할 수 있습니다.

Router 화면을 가리키며:

> 먼저 Router가 요청을 적치지시 생성 Intent로 분류하고, 입고번호와 Location을 구조화된 파라미터로 추출했습니다.

Tool Executor 화면을 가리키며:

> Tool Executor는 LLM이 만든 임의 코드를 실행하지 않습니다. 등록된 Intent Handler만 선택하고, 여기서는 실제 변경 대신 Draft만 생성했습니다.

Approval Gate 화면을 가리키며:

> 마지막 Approval Gate가 이 요청을 승인 필요 작업으로 판정한 것과 Draft ID까지 확인할 수 있습니다.

추천 Trace의 RAG 단계를 가리키며:

> 정책 근거가 필요한 요청에서는 검색 결과를 바로 답변에 사용하지 않습니다. PRISM으로 후보를 재평가하고, Sufficient Context 게이트가 현재 근거만으로 답변 가능한지 검증합니다. 근거가 부족하면 질의를 다시 작성해 최대 두 번 검색하고, 그래도 부족하면 답변을 지어내지 않고 abstain합니다.

토큰·호출 수를 가리키며:

> 요청별 LLM 호출 수와 토큰도 저장하므로 품질뿐 아니라 비용과 지연의 회귀까지 추적할 수 있습니다.

**AI 구축 관점 핵심 메시지**

- LangGraph 실행 경로가 실제 화면에서 관측됨
- LLM 판단, 코드 계산, RAG 근거, 승인 판정이 분리됨
- 결과가 아니라 판단 과정과 실행 계약을 검증 가능
- 토큰과 호출 수까지 포함한 운영형 Observability

**표현상 주의**

이 화면은 모델의 비공개 내부 Chain-of-Thought를 노출하는 화면이라고 설명하지 않는다. 업무적으로 필요한 노드별 입력·출력, Tool 결과, 검색 근거와 승인 판정을 기록한 실행 Trace라고 설명한다.

### 02:35~03:10 — Approval에서 Dry-run 검토 및 실제 실행

**화면 조작**

1. `Approval` 탭으로 이동한다.
2. 승인 대기 목록에서 INB003 적치지시 Draft를 선택한다.
3. 작업 유형, 대상, Location과 Dry-run 정보를 보여준다.
4. `승인` 버튼을 누른다.
5. 성공 토스트와 처리 결과를 보여준다.
6. 가능하면 실행 완료 목록 또는 운영 데이터의 `stocking_tasks`에서 반영 결과를 짧게 확인한다.

**내레이션**

> 생성된 Draft는 Approval 화면에 모입니다. 운영자는 실행 대상과 변경 내용을 Dry-run으로 확인한 뒤 승인하거나 거절할 수 있습니다.

승인 버튼을 누르며:

> 승인을 누른 시점에만 실제 WMS 상태가 변경됩니다. Draft는 승인 대기, 승인, 실행 완료의 상태 전이로 관리되며, 이미 처리된 Draft는 다시 승인할 수 없습니다.

**강조 포인트**

- 추천에서 실제 Action까지 하나의 서비스 흐름으로 연결
- 사용자 통제권 보장
- 승인 이력과 실행 결과 추적
- PoC 챗봇이 아닌 실행 가능한 업무 서비스

### 03:10~04:05 — Blackboard 멀티에이전트 자동운영

**권장 준비 상태**

- 촬영 전에 자동운영 이벤트와 실행 로그가 최소 1건 이상 존재하도록 준비한다.
- DES 워밍업을 미리 완료한다.
- 실시간 요청, Dispatch 계산, Zone Route와 Action 실행순서 로그가 화면에 표시되는지 확인한다.

**화면 조작**

1. `창고 자동운영` 탭으로 이동한다.
2. 상단에서 도메인 에이전트 상태를 보여준다.
3. 실시간 요청 목록에서 한 건을 선택한다.
4. 선택 요청의 생애주기를 왼쪽에서 오른쪽 또는 위에서 아래로 따라가며 보여준다.
5. `작업 배정 계산 로그`에서 `dispatch_score`와 배정 사유를 보여준다.
6. `ZONE 방문순서 계산 로그`에서 closed TSP 경로를 보여준다.
7. `액션 실행 순서 로그`에서 다음 순서를 가리킨다.

```text
FINISH_ZONE_LEG → START_ZONE_WORK → ALLOCATE_TEAM
```

8. 시간이 허용되면 `1회 실행`을 눌러 로그가 추가되는 모습을 보여준다.

**내레이션**

> 대화형 Agent와 별도로 Blackboard 멀티에이전트가 창고를 주기적으로 자동 운영합니다. 입고, 적치, 피킹, 출고, 자원 에이전트가 이벤트를 보고 Action을 제안하고, Control Loop가 우선순위와 정책, 락, 자원 조건을 검증해 실행합니다.

요청 생애주기를 가리키며:

> 각 요청은 이벤트 수집부터 Action 제안, 사전검증, 실행과 완료까지 생애주기로 추적됩니다.

Dispatch 로그를 가리키며:

> 작업팀 배정은 단순 FIFO가 아니라 마감 긴급도, 대기시간, 예상 작업시간과 이동 요소를 반영한 dispatch score로 결정됩니다.

Zone Route를 가리키며:

> 피킹은 입구에서 출발해 필요한 Zone을 방문하고 다시 입구로 돌아오는 closed TSP 경로를 계산합니다.

실행순서 로그를 가리키며:

> 또 완료 작업으로 자원을 먼저 해제하고, 대기 작업을 시작한 뒤 신규 팀을 배정해 자원 교착과 불필요한 대기를 줄입니다. 이 계산과 실행순서 역시 모두 로그로 남습니다.

**AI 구축 관점 강조**

- 대화형 Agent와 자율 운영 Agent의 공존
- 이벤트 기반 Blackboard 패턴
- LLM 제안 이후 Policy·Lock·Executor 게이트
- 휴리스틱과 최적화 결과의 설명 가능성

**서비스 관점 강조**

- 운영자가 반복적으로 작업 순서를 판단하는 부담 감소
- 특정 작업이 먼저 처리된 이유를 사후 검증 가능
- 자동운영 상태와 장애 지점을 한 화면에서 확인

### 04:05~04:45 — What-if 시뮬레이션과 디지털 트윈

**화면 조작**

1. `Warehouse Simulation` 탭으로 이동한다.
2. 이미 생성된 Baseline을 선택하거나 화면에 표시한다.
3. 작업자 증원 조건을 `+1`로 설정한다.
4. What-if 실행 버튼을 누른다.
5. Baseline과 What-if KPI 비교를 보여준다.
6. 출고지연, 피킹 대기시간, 작업팀 가동률 등 화면에서 실제 변화가 큰 지표를 가리킨다.
7. 하단의 2D 디지털 트윈과 Event Timeline을 짧게 보여준다.

## 4. 촬영 전 준비 체크리스트

## 5. 장애 발생 시 대체 진행안

| 문제 | 대체 진행 방법 |
| --- | --- |
| LLM 응답 지연 | 미리 생성한 동일 세션을 좌측에서 선택하고 결과 및 Trace 설명으로 전환 |
| 추천 Location이 달라짐 | 화면에 표시된 실제 추천 Location을 다음 상태 변경 질문에 사용 |
| RAG 근거 부족 | 사전 생성한 `왜 Zone A를 추천했어?` Trace를 AI 관측에서 선택 |
| Draft가 이미 존재함 | Approval 탭의 기존 Draft를 사용하고 중복 방지 기능으로 설명 |
| 자동운영 신규 로그 없음 | 기존 요청을 선택해 생애주기·Dispatch·Route·실행순서 로그 설명 |
| DES 실행 지연 | 사전 생성한 Baseline과 What-if 버전을 비교 대상으로 선택 |
| 5분 초과 예상 | 운영 데이터 확인을 생략하고 Approval 성공 토스트까지만 표시 |

## 6. 발표에서 반복할 핵심 문장

아래 표현 중 세 문장 이상을 영상에 포함한다.

1. **LLM은 의도와 파라미터를 해석하고, 실제 계산과 실행은 등록된 코드가 담당합니다.**
2. **근거가 필요한 답변은 PRISM 재평가와 충분성 검증을 통과하며, 근거가 부족하면 답변하지 않습니다.**
3. **고위험 상태 변경은 Draft와 Approval Gate를 거쳐야 실제 WMS에 반영됩니다.**
4. **AI 관측 화면에서 Intent, Tool 결과, RAG 근거, 승인 판정, 토큰과 호출 수까지 확인할 수 있습니다.**
5. **자동운영의 배정 점수, TSP 경로와 실행순서를 모두 저장해 사후 검증할 수 있습니다.**
6. **현재 상태를 DES에 연결해 자원 투입 효과를 실제 변경 전에 비교할 수 있습니다.**
